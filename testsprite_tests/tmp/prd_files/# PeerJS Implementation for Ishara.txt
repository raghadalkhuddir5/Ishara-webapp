# PeerJS Implementation for Ishara

## Overview

This implementation replaces Agora with PeerJS for WebRTC video calling in the Ishara application. PeerJS simplifies WebRTC by providing:

- **Built-in signaling server** (no need for custom Firestore signaling)
- **Simplified peer connection handling** (work with peer IDs instead of raw offers/answers)
- **Free development server** (PeerJS Cloud)
- **Easy deployment** (can host your own PeerServer)

## Architecture

### 1. PeerJS Service (`src/services/peerjsService.ts`)

The main service that handles:
- Peer initialization and connection
- Media stream management (audio/video)
- Screen sharing
- Data channel communication
- Call controls (mute, camera toggle, etc.)

### 2. Call Service (`src/services/callService.ts`)

Updated to work with PeerJS:
- Manages call lifecycle in Firestore (call metadata, duration, quality metrics)
- Tracks call records for sessions (separate from peer connection logic)
- Note: Peer IDs are NOT stored in Firestore - they are derived deterministically from session ID and role

### 3. Video Call Component (`src/components/VideoCall.tsx`)

Completely rewritten to use PeerJS:
- Initializes PeerJS peer on mount
- Handles caller/callee logic
- Manages media streams and UI controls
- Provides real-time call status

## How It Works

### Call Flow

1. **Session Confirmation**: When an interpreter accepts a session, the status is updated to "confirmed"

2. **Peer ID Generation**: 
   - Both participants derive peer IDs deterministically from session ID:
     - Deaf/mute user: `${sessionId}-caller`
     - Interpreter: `${sessionId}-callee`
   - No Firestore storage needed for peer IDs (computed on-the-fly)

3. **Call Initiation**:
   - Deaf/mute user (caller) initializes PeerJS with their deterministic peer ID
   - Interpreter (callee) initializes PeerJS with their deterministic peer ID
   - Deaf/mute user calls interpreter's peer ID (`${sessionId}-callee`)
   - Interpreter receives the call automatically via PeerJS listeners

4. **Media Exchange**: WebRTC handles peer-to-peer media streaming

### Firestore Schema

**Note**: Peer IDs are NOT stored in Firestore. They are computed deterministically:
- Caller peer ID: `${sessionId}-caller`
- Callee peer ID: `${sessionId}-callee`

The actual `calls` collection stores call metadata:

```typescript
interface CallData {
  call_id?: string;              // Document ID
  session_id: string;            // Reference to session
  user_id: string;               // Deaf/mute user UID
  interpreter_id: string;         // Interpreter UID
  started_at: Timestamp;
  ended_at?: Timestamp;
  duration_seconds?: number;
  status: 'active' | 'ended' | 'failed';
  recording_url?: string;
  recording_duration?: number;
  call_quality: {
    video_quality: 'poor' | 'fair' | 'good' | 'excellent';
    audio_quality: 'poor' | 'fair' | 'good' | 'excellent';
    connection_stability: 'poor' | 'fair' | 'good' | 'excellent';
  };
  technical_metrics?: {
    avg_bitrate: number;
    packet_loss: number;
    jitter: number;
    latency: number;
  };
  created_at: Timestamp;
}
```

## Migration from Agora

### What Changed
- ❌ Removed Agora SDK dependency from main app (`package.json` - frontend)
- ⚠️ **Note**: Agora token generation functions still exist in `functions/` directory (Cloud Functions) but are not used by the frontend
- ❌ Removed token generation logic from frontend
- ❌ Removed channel-based calling
- ✅ Added PeerJS SDK (`peerjs: ^1.5.5` and `@types/peerjs: ^0.0.30`)
- ✅ Added deterministic peer ID-based calling (computed from session ID)
- ✅ Simplified signaling (no custom Firestore signaling - PeerJS handles it)
- ✅ Peer IDs are computed, not stored in Firestore

### Benefits
- **Simpler**: No token management
- **Free**: No Agora costs
- **Flexible**: Can host your own signaling server
- **Reliable**: PeerJS handles connection management

## Troubleshooting

### Common Issues

1. **"Peer not initialized"**
   - Check if PeerJS server is accessible
   - Verify network connection

2. **"No remote stream"**
   - Ensure both participants have joined
   - Check browser permissions for camera/microphone

3. **"Call failed"**
   - Verify both participants have initialized their PeerJS peers
   - Check that peer IDs match the expected format (`${sessionId}-caller` or `${sessionId}-callee`)
   - Ensure PeerJS server is accessible (PeerJS Cloud or custom server)
   - Check console for detailed error messages

### Debug Mode

Enable debug logging by opening browser console. The implementation provides detailed logs for:
- Peer initialization
- Call establishment
- Media stream handling
- Error conditions
